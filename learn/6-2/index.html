STATIONS 01-06 - DATA FORMATS

CSV, JSON, XML, Binary & Fixed-Width

Know the format. Know why it matters. Know what the examiner is looking for.

FORMAT 01
📊CSV
Comma-Separated Values
Plain text format where rows are records and values are separated by commas. Simple, universal, and readable by almost every application - the default for data exchange and bulk transfers.
OrderID,Customer,Total\nORD-001,S.Mitchell,38.97\nORD-002,J.Thornton,49.00
Human-readableLightweightSpreadsheetsNo nesting
FORMAT 02
🔗JSON
JavaScript Object Notation
Lightweight text format using key-value pairs and arrays. The dominant format for APIs and web services - structured, human-readable, and natively supported by JavaScript and most languages.
{"order":{\n "id":"ORD-001",\n "customer":"S.Mitchell",\n "total":38.97\n}}
APIsWeb servicesNested dataJavaScript
FORMAT 03
🏷XML
eXtensible Markup Language
Tag-based hierarchical format for structured data. Verbose but extremely flexible - used in enterprise systems, config files, document formats (Word, XHTML). Supports schema validation via XSD.
<order>\n <id>ORD-001</id>\n <total>38.97</total>\n</order>
VerboseEnterpriseSelf-describingSchemas
FORMAT 04
⚙Binary
Binary / Machine-Readable Format
Data stored as raw bytes (0s and 1s) - not human-readable but extremely compact and fast. Used for images, audio, executables, video, and high-performance database storage formats.
4F 52 44 2D 30 30 31\n53 2E 4D 69 74 63 68\n65 6C 6C 00 26 E5 47
CompactFastNot readableMedia files
FORMAT 05
📏Fixed-Width
Fixed-Width / Positional Format
Each field occupies a fixed number of characters - no delimiters. Common in legacy banking, government, and mainframe systems where field position defines meaning. Strict but fast to parse.
ORD-001S.Mitchell 38.97\nORD-002J.Thornton 49.00\n|--7--||---12-----||--6--|
LegacyBankingNo delimiterPositional
EXAMINER GUIDE
🎯Choose Right
Format selection = marks
Examiners ask which format is most suitable and why - the answer always needs a reason linked to the use case.
Spreadsheet / bulk transfer -> CSV
Web API / app -> JSON
Enterprise / config -> XML
Media / performance -> Binary
Banking / mainframe -> Fixed-width

FLIP TO REVEAL
A developer is building a REST API to serve product data to a mobile app. Which format should they use and why?
tap to flip
ANSWER
JSON. It is lightweight, natively parsed by JavaScript, supports nested objects, and is the industry standard for web APIs - minimising bandwidth and maximising compatibility.
FLIP TO REVEAL
What is the main disadvantage of XML compared to JSON for API use?
tap to flip
ANSWER
XML is verbose - opening and closing tags for every field significantly increase file size. For APIs this means higher bandwidth costs and slower transfer compared to equivalent JSON.
FLIP TO REVEAL
Why would a bank use fixed-width format instead of CSV for transaction files?
tap to flip
ANSWER
Fixed-width uses positional parsing - programs read bytes at fixed offsets rather than scanning for delimiters. This is faster on mainframe systems and avoids issues if data contains commas.
FLIP TO REVEAL
Why is binary format used for storing images rather than XML or CSV?
tap to flip
ANSWER
Binary stores pixel data as raw bytes - far more compact than text encoding. A 4K image as XML would be unworkably large. Binary enables fast read/write and efficient compression.
FLIP TO REVEAL
A business wants to export 50,000 customer records to give to a partner company. Which format is most appropriate?
tap to flip
ANSWER
CSV. It is universally supported by Excel, Python, R, and SQL import tools. For flat tabular data with no nesting, CSV is the pragmatic, universally compatible choice.
FLIP TO REVEAL
Name one format that supports nested data and one that does not. Explain the business implication.
tap to flip
ANSWER
JSON supports nesting (one order containing multiple items). CSV does not - each row must be flat. For complex relational data, CSV requires multiple files, increasing processing complexity.

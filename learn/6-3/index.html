Data Wrangling, Business Tools & Data Models

Understand how data is managed, organised and used for business — and how to explain it for full marks.

STATION 01 — NAME IT
📋
Data Wrangling (6.3.1)
Data wrangling is the process of transforming raw data into a usable format. The core stages are: structure (organise into tables/fields), clean (remove errors/duplicates), enrich (add context or merge sources), validate (check accuracy) and output (export for use).
HOSPITAL SCENARIO
NorthGen Trust is merging 14,000 patient records from two legacy systems — PAS-Alpha and CareLink. Data wrangling is applied to structure fields (e.g. DOB formats differ), clean duplicates, enrich with GP postcode data, validate NHS numbers, and output a unified CSV for the new EPR system.
StructureCleanEnrichValidateOutput
STATION 01b — NAME IT
⚙
Core Data System Functions
All data systems share six core functions: Input (data entry), Search (query/retrieve), Save (store/persist), Integrate (link systems), Organise (structure/sort), Output (report/export). The feedback loop checks outputs against original inputs to maintain integrity.
HOSPITAL SCENARIO
The merged EPR system inputs patient demographics, searches by NHS number, saves securely to a database, integrates with pharmacy and lab systems, organises by ward, and outputs discharge summaries. The feedback loop flags records where post-merge data fails NHS number validation.
InputSearchSaveIntegrateOrganiseOutput
STATION 01c — NAME IT
📈
Data Visualisation
Visualisation presents data graphically to aid understanding. Common types: graphs (trends over time), charts (comparisons), data tables (raw structured data), reports (formal summaries), infographics (visual narratives for non-technical audiences).
HOSPITAL SCENARIO
After the data merge, the trust produces: a line graph showing patient admissions over 12 months, a bar chart comparing readmission rates by ward, a data table of all 14,000 migrated records, a management report on data quality, and an infographic for the board showing key statistics from the migration.

How to earn Name marks — CA6.3
▷ EXAM QUESTION [3 MARKS]

A hospital trust is preparing to merge patient records from two legacy systems. Describe the process of data wrangling. [3 marks]

⚠ WEAK ANSWER — 1/3 marks

"Data wrangling is when you clean up data to make it usable. You remove errors and duplicates."

✖ Only names the clean stage. Missing structure, enrich, validate and output. No technical terminology for the other stages earns 1 mark maximum.
✓ STRONG ANSWER — 3/3 marks

"Data wrangling is the process of preparing raw data for use through five stages: structuring (organising fields into a consistent format), cleaning (removing errors and duplicates), enriching (adding context by merging with additional sources), validating (checking accuracy against rules) and outputting (exporting the processed data for use)."

✓ Names all five stages with technical terms. Each stage correctly identified = marks awarded.
✉ NAME-IT TECHNIQUE — EXAMINER TIP

For CA6.3 Name marks: use the exact spec terminology. The examiner's mark scheme lists specific terms — "structure", "clean", "enrich", "validate", "output" (wrangling), "conceptual", "logical", "physical" (data models), "BI software", "CRM", "financial planning" (business tools). Using a synonym instead of the spec term risks losing the mark.

STATION 02 — EXPLAIN IT
📚
Business Information Tools (6.3.2)
BI Software (e.g. Power BI, Tableau) turns data into interactive dashboards and insights for decision-making. Financial planning & analysis tools (e.g. Adaptive Insights) model budgets and forecasts. CRM (Customer Relationship Management, e.g. Salesforce) manages customer data, interactions and pipelines.
EXPLAIN THE IMPACT
A BI tool enables the trust’s management team to drill down into merged patient data and identify trends, e.g. which wards have the highest readmission rates. This means senior managers can allocate resources more effectively, reducing costs and improving patient outcomes — a direct business benefit from the data system.
BI SoftwareFinancial PlanningCRMDashboards
STATION 03 — IMPACT IT
📄
Data Models (6.3.3)
A conceptual model defines what data exists (entities and relationships) at a high level for stakeholders. A logical model defines how data is structured (attributes, keys, normalisation) independent of technology. A physical model defines how data is stored in a specific DBMS (tables, indexes, data types). A hierarchical model organises data in a tree (parent→child). A relational model uses tables linked by foreign keys.
IMPACT ON THE BUSINESS
Before the hospital data merge, architects produced a conceptual model (entities: Patient, Ward, Consultant), then a logical model (Patient has NHS_Number PK, DOB, Ward_ID FK), then a physical model (SQL Server tables, indexes on NHS_Number for fast lookup). The relational model allowed the two legacy databases to be merged by linking on NHS numbers — making the 14,000-record merge achievable in a weekend rather than months of manual work.

How Business Information Tools Work
BI software (e.g. Power BI, Tableau) works by connecting to data sources, processing queries and rendering interactive dashboards. The pipeline is: ingest data → transform → model → visualise. Financial planning tools (e.g. Adaptive Insights) work by building forecast models from historical data, allowing managers to run “what-if” scenarios. ⚙
How Data System Functions Work
Each core function has a specific mechanism: Input works by accepting data via forms, APIs or sensors and writing to a storage layer. Search works by executing a query (e.g. SQL SELECT) against indexed fields, returning matching records. Save works by serialising data and writing it to persistent storage with a transaction log for recovery. Integrate works by mapping fields between systems using an API or ETL pipeline, resolving schema differences. Organise works by sorting, grouping or categorising records using defined rules. Output works by formatting stored data into reports, exports or API responses for consumers.
HOSPITAL SCENARIO — EXPLAIN THE MECHANISM
When a nurse inputs a patient observation, the Input function writes a timestamped record to the observations table. The Integrate function then maps this to the patient’s NHS number, linking it to the main record via a foreign key — enabling the Search function to retrieve all observations for a given patient in milliseconds using an indexed query.

span>STATION 02c — EXPLAIN IT
📋
How Data Models Enable Organisation
A conceptual model works by defining entities and relationships as a diagram (ERD), giving stakeholders a shared vocabulary before any technical work begins. A logical model works by translating entities into tables with attributes and keys, independent of any database platform, so developers can validate structure. A physical model works by specifying exact data types, indexes and constraints in a target DBMS (e.g. SQL Server), enabling storage optimisation. A relational model works by using primary/foreign key pairs to link tables, so a single query can JOIN multiple tables and retrieve related data without duplication. A hierarchical model works by nesting child records under a parent, enabling fast read access for tree-structured data (e.g. an org chart).
HOSPITAL SCENARIO — EXPLAIN THE MECHANISM
The relational model works by linking the Patient table (NHS_Number PK) to the Ward table (Ward_ID FK) and the Admission table (NHS_Number FK, Ward_ID FK). A JOIN query can retrieve all patients in a given ward with one SQL statement — without duplicating patient demographic data across multiple tables, which would create update anomalies.

How to earn Explain marks — CA6.3
The Explain mark is earned by describing the MECHANISM — not repeating the name.

Weak answer (Name only): “The organisation uses a BI tool.” ⚠ No explain mark — this just names the tool.

Strong answer (Explain mark): “The BI tool works by connecting to the data warehouse and executing aggregate queries, which are rendered as interactive dashboards so managers can identify trends at ward level.” ✅

Common student errors on the Explain mark:
1. Saying “it helps the business” instead of explaining the technical process
2. Repeating the definition instead of the mechanism
3. Using vague phrases like “it processes data” without saying HOW
4. Confusing Explain with Impact (explaining a business outcome rather than a technical process)
3-MARK STRUCTURE — N+E+I
[N] “NorthGen Trust uses a relational data model.” [E] “This works by linking the Patient table to the Admission table via a foreign key (NHS_Number), so a single SQL JOIN retrieves all admissions for a patient without duplicating demographic data.” [I] “This means the merged 14,000-record database can be queried in real time, enabling clinical staff to access patient history instantly — reducing decision-making delays and improving patient safety.”

How does a relational model prevent data duplication?
📲 tap to flip
By storing each entity once in its own table and linking related records via primary/foreign key pairs. A JOIN query retrieves combined data at query time, so demographic data is never duplicated across rows.

📲 tap to flip
A logical model defines structure (tables, keys, attributes) platform-independently. A physical model translates this into a specific DBMS — specifying data types (e.g. VARCHAR(50)), indexes for performance and storage constraints.

IMPACT IT — State the business consequence of the data model choice

STATION 03a

Impact of Data Model Choice

The Impact mark explains the business consequence of the data structure chosen. It answers: "So what does this mean for the organisation?"

Relational modelenables real-time multi-table queries → faster management reporting
Hierarchical modelallows fast reads of fixed paths → suitable for high-volume transaction processing
Physical modeloptimises DBMS storage layout → reduces I/O costs and improves response times
Conceptual modelcreates shared vocabulary across teams → reduces miscommunication during development
KEY RULE The Impact mark must name a stakeholder or organisational outcome — not just restate the Explain mark.

I STATION — TECHNIQUE

Sentence Starters & Failure Modes

The Impact mark is the core differentiator of the MSM methodology. Many students lose this mark by repeating the Explain mark instead of stating the business outcome.

Starter: "This means…"…the finance director can view live cash flow without waiting for manual reports
Starter: "As a result…"…stock managers can identify shortfalls before they affect production
Starter: "This enables…"…customer service agents to resolve complaints faster, improving satisfaction scores
Starter: "The impact is…"…reduced data entry errors, cutting the cost of corrections for the operations team
COMMON FAILURE MODES
  • Vague statements: "it is better" or "it improves things" — no named stakeholder or outcome
  • Repeating Explain: restating how the model works instead of what it achieves for the business
  • No named stakeholder: saying "users" instead of "the warehouse manager" or "the finance team"
  • Circular logic: "the relational model impacts the relations between tables" — this is just the Name mark repeated

What is the Impact mark testing?

The business consequence of the data model or tool choice — the "so what?" for the organisation or a named stakeholder.

Name four Impact sentence starters.

"This means…" / "As a result…" / "This enables…" / "The impact is…" — always followed by a named stakeholder or measurable outcome.

What is the most common failure mode for the Impact mark?

Repeating the Explain mark — describing how the model works again instead of stating what it achieves for the business.

What business impact does a relational model provide?

Enables real-time multi-table queries → management can access joined reports instantly, reducing time spent on manual data consolidation.

What business impact does a CRM system provide?

Centralises customer interaction history → sales teams can identify upsell opportunities and improve retention rates without manual record checking.

Why must the Impact mark name a stakeholder?

Mark schemes award the Impact mark only when a specific organisational role or group is identified — generic phrases like "it is better" score zero.

AXIOM-7

The Impact Paradox

A student argues: "If I write a really detailed Explain mark, the examiner should give me the Impact mark too — because the consequence is implied."

Using your understanding of the MSM methodology, explain why this argument fails and what the student must do differently to secure the Impact mark.

AXIOM-7 — DEBRIEF

Model Answer: The argument fails because examiners award marks only for what is explicitly written, not what is implied. Even a sophisticated Explain mark does not earn the Impact mark unless the student states a named business consequence or stakeholder outcome. The MSM methodology requires the student to add a distinct sentence using a trigger phrase such as "This means…" or "As a result…" followed by a specific organisational benefit — for example, "This means the warehouse manager can track stock levels in real time, preventing costly overstocking." Implication is not evidence; explicit statement is required.
